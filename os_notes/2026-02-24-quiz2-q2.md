## Key Concepts to Understand

### User-Level Threading

> **💡 Hint:** User-level threads are managed entirely in user space without kernel support, making them lightweight but requiring careful implementation of context switching.

**What it means:**
- Threads are scheduled by user-space code, not the kernel
- Context switching happens without system calls
- Multiple threads share a single kernel-level process

### Thread Context Switching

The critical operation for threads is saving and restoring register state:

| Component | Purpose |
|-----------|---------|
| **Stack Pointer (sp)** | Points to thread's stack |
| **Program Counter (pc)** | Resumption point |
| **General Registers** | Saved computation state |
| **Frame Pointer (fp)** | Stack frame management |

### Synchronization Primitives

> **⚠️ Warning:** Without proper locking, concurrent access to shared data structures will cause race conditions and data corruption.

**Essential mechanisms:**
- **Spinlock**: Busy-waits until lock is free (suitable for short critical sections)
- **Mutex**: Sleep-based lock (suitable for longer critical sections)
- **Condition Variables**: Allow threads to wait for specific conditions

## Common Implementation Patterns

### Thread Structure Definition

```c
struct thread {
  char       *stack;        // The stack for this thread
  int         context;      // Index of saved context (in struct proc)
  int         state;        // FREE, RUNNABLE, RUNNING, BLOCKED
};
```

### Thread Switching Assembly Pattern

```asm
# Save current thread context
# Registers: a0=old context pointer, a1=new context pointer
sd ra, 0(a0)      # Save return address
sd sp, 8(a0)      # Save stack pointer
sd s0, 16(a0)     # Save frame pointer
# ... save more registers ...

# Restore new thread context
ld ra, 0(a1)      # Load return address
ld sp, 8(a1)      # Load stack pointer
ld s0, 16(a1)     # Load frame pointer
# ... restore more registers ...
ret
```

### Basic Lock Implementation

```c
struct lock {
  uint locked;  // Is the lock held?
  char *name;   // Name of lock
};

void acquire(struct lock *lk) {
  while (__sync_lock_test_and_set(&lk->locked, 1) != 0)
    ;  // Spin until acquired
}

void release(struct lock *lk) {
  __sync_lock_release(&lk->locked);
}
```

## Common Pitfalls

> **⚠️ Warning:** These are frequent sources of bugs:

1. **Insufficient Context Saving**
   - Missing register saves causes corruption
   - Solution: Save all callee-saved and caller-saved registers

2. **Race Conditions in Lock Code**
   - Non-atomic test-and-set operations
   - Solution: Use atomic instructions

3. **Stack Overlap**
   - Threads using overlapping stack space
   - Solution: Allocate separate stacks far apart

4. **Lock Deadlocks**
   - Threads waiting for each other's locks
   - Solution: Establish lock ordering discipline

5. **Busy-Wait Efficiency**
   - Spinlocks waste CPU
   - Solution: Use sleep-based locks for long waits

## Debugging Tips

### Inspecting Thread State
- Print thread structure contents
- Verify stack pointers are in valid range
- Check context register values

### Verifying Context Switches
- Add timestamps to trace execution
- Verify PC (program counter) changes correctly
- Confirm stack pointer switches between calls

### Testing Lock Behavior
- Use atomic counters to verify mutual exclusion
- Add lock held flags
- Verify proper release of locks
