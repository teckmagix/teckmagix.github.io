## Part 1: User-Level Threads (`uthread`)

### What Is `uthread`?

xv6's `uthread` is a **cooperative, user-level threading library**. It does NOT use the kernel scheduler — instead, threads voluntarily yield control to each other via `thread_schedule()`.

### Key Files

| File | Purpose |
|---|---|
| `user/uthread.c` | Thread creation, scheduling, and switching logic |
| `user/uthread_switch.S` | Assembly routine that saves/restores registers (context switch) |

---

### `user/uthread.c` — Explained

```c
// The thread structure holds each thread's state
struct thread {
  char       stack[STACK_SIZE]; // each thread has its own stack
  int        state;             // FREE, RUNNING, or RUNNABLE
  struct context ctx;           // saved registers (YOUR TASK)
};
```

#### Thread States

| State | Meaning |
|---|---|
| `FREE` | Slot is unused |
| `RUNNING` | Currently executing |
| `RUNNABLE` | Ready to run, but not currently executing |

#### `thread_create()`

```c
void thread_create(void (*func)())
{
  struct thread *t;

  for (t = all_thread; t < all_thread + MAX_THREAD; t++) {
    if (t->state == FREE) break;
  }
  t->state = RUNNABLE;

  // YOUR TASK: Set up the thread's context so that
  // thread_switch will jump to `func` when first scheduled.
  // Specifically, set t->ctx.ra = (uint64)func;
  // and set t->ctx.sp = (uint64)(t->stack + STACK_SIZE);
}
```

> **💡 Hint:** The stack pointer must point to the **top** of the stack (stacks grow downward in RISC-V), so use `t->stack + STACK_SIZE`.

#### `thread_schedule()`

```c
void thread_schedule(void)
{
  struct thread *t, *next_thread;

  // Find the next RUNNABLE thread (not the current one)
  next_thread = 0;
  t = current_thread + 1;
  for(int i = 0; i < MAX_THREAD; i++){
    if(t >= all_thread + MAX_THREAD)
      t = all_thread;
    if(t->state == RUNNABLE) {
      next_thread = t;
      break;
    }
    t = t + 1;
  }

  if (next_thread == 0) {
    printf("thread_schedule: no runnable threads\n");
    exit(-1);
  }

  if (current_thread != next_thread) {
    next_thread->state = RUNNING;
    t = current_thread;
    current_thread = next_thread;
    // YOUR TASK: call thread_switch(&t->ctx, &next_thread->ctx)
    thread_switch((uint64)&t->ctx, (uint64)&next_thread->ctx);
  }
}
```

### `user/uthread_switch.S` — The Context Switch

This is the **core assembly routine** you must implement. It saves the current thread's callee-saved registers onto its stack/context and restores the next thread's saved registers.

#### RISC-V Callee-Saved Registers

These are the registers that a function must preserve (save and restore) per the RISC-V calling convention:

| Register | ABI Name | Role |
|---|---|---|
| x1 | ra | Return address |
| x2 | sp | Stack pointer |
| x8 | s0/fp | Frame pointer |
| x9 | s1 | Saved register |
| x18–x27 | s2–s11 | Saved registers |

#### Required `struct context` Layout

```c
struct context {
  uint64 ra;   // return address — where thread resumes
  uint64 sp;   // stack pointer

  // callee-saved registers
  uint64 s0;
  uint64 s1;
  uint64 s2;
  uint64 s3;
  uint64 s4;
  uint64 s5;
  uint64 s6;
  uint64 s7;
  uint64 s8;
  uint64 s9;
  uint64 s10;
  uint64 s11;
};
```

#### `thread_switch` Implementation (Assembly)

```asm
    .text

    /*
     * void thread_switch(struct context *old, struct context *new);
     *
     * a0 = pointer to old context (save here)
     * a1 = pointer to new context (restore from here)
     */
    .globl thread_switch
thread_switch:
    /* Save callee-saved registers into old context (a0) */
    sd ra,   0(a0)
    sd sp,   8(a0)
    sd s0,  16(a0)
    sd s1,  24(a0)
    sd s2,  32(a0)
    sd s3,  40(a0)
    sd s4,  48(a0)
    sd s5,  56(a0)
    sd s6,  64(a0)
    sd s7,  72(a0)
    sd s8,  80(a0)
    sd s9,  88(a0)
    sd s10, 96(a0)
    sd s11,104(a0)

    /* Restore callee-saved registers from new context (a1) */
    ld ra,   0(a1)
    ld sp,   8(a1)
    ld s0,  16(a1)
    ld s1,  24(a1)
    ld s2,  32(a1)
    ld s3,  40(a1)
    ld s4,  48(a1)
    ld s5,  56(a1)
    ld s6,  64(a1)
    ld s7,  72(a1)
    ld s8,  80(a1)
    ld s9,  88(a1)
    ld s10, 96(a1)
    ld s11,104(a1)

    ret   /* jump to ra (the new thread's return address / entry point) */
```

> **⚠️ Warning:** Only save **callee-saved** registers (`ra`, `sp`, `s0–s11`). Do NOT save `a0–a7`, `t0–t6` — those are caller-saved and the caller is responsible for them.

### Setting Up a New Thread's Context

When `thread_create` creates a new thread, the thread has **never run before**. We need to fake a context so that when `thread_switch` restores it, execution begins at `func`.

```c
void thread_create(void (*func)())
{
  struct thread *t;
  for (t = all_thread; t < all_thread + MAX_THREAD; t++)
    if (t->state == FREE) break;

  t->state = RUNNABLE;

  // Initialize context
  memset(&t->ctx, 0, sizeof(t->ctx));
  t->ctx.ra = (uint64)func;                    // return address = function to run
  t->ctx.sp = (uint64)(t->stack + STACK_SIZE); // top of stack (grows down)
}
```

#### Why `stack + STACK_SIZE`?

```
Low address   [ thread->stack[0] ]    <- bottom of array
              [      ...        ]
High address  [ thread->stack[STACK_SIZE - 1] ]  <- top, sp starts here
```

Stacks grow **downward** (toward lower addresses). The stack pointer starts at the highest address.

---

### Complete Execution Flow

```
thread_create(func)
  → sets ctx.ra = func, ctx.sp = top of stack

thread_schedule()
  → finds next RUNNABLE thread
  → calls thread_switch(&old->ctx, &new->ctx)

thread_switch (assembly)
  → saves current registers into old->ctx
  → loads registers from new->ctx
  → ret  ← jumps to new->ctx.ra (= func for first run)

func() runs...
  → calls thread_yield() when it wants to give up CPU

thread_yield()
  → sets current_thread->state = RUNNABLE
  → calls thread_schedule()
```

## Part 2: POSIX Mutex Locks (`pthread_locks`)

### Overview

`notxv6/pthread_locks.c` benchmarks two types of locks:

| Lock Type | Description |
|---|---|
| **Mutex** | OS-assisted sleep lock (`pthread_mutex_t`) |
| **Spin lock** | Busy-wait lock using `__sync_lock_test_and_set` |

### Mutex Implementation

```c
pthread_mutex_t lock;            // declare a mutex

pthread_mutex_init(&lock, NULL); // initialize

// Critical section:
pthread_mutex_lock(&lock);
// ... protected code ...
pthread_mutex_unlock(&lock);
```

### Spin Lock Implementation

```c
struct spinlock {
  int locked;
};

void acquire(struct spinlock *lk) {
  // Atomically set locked = 1; spin until we succeed
  while (__sync_lock_test_and_set(&lk->locked, 1) != 0)
    ; // spin / busy-wait
  __sync_synchronize(); // memory barrier
}

void release(struct spinlock *lk) {
  __sync_synchronize(); // memory barrier
  __sync_lock_release(&lk->locked); // atomically set locked = 0
}
```

### Key Atomic Primitives

| Function | Description |
|---|---|
| `__sync_lock_test_and_set(&x, 1)` | Atomically sets `x=1`, returns old value |
| `__sync_lock_release(&x)` | Atomically sets `x=0` |
| `__sync_synchronize()` | Full memory barrier (prevents reordering) |

> **💡 Hint:** A spin lock is faster for **short critical sections** and **low contention**. A mutex is better when threads wait a **long time** (it sleeps instead of burning CPU).

### Mutex vs Spin Lock Comparison

| Feature | Mutex | Spin Lock |
|---|---|---|
| Waiting behavior | Sleeps (OS schedules another thread) | Busy-waits (burns CPU cycles) |
| Context switch overhead | Yes | No |
| Good for | Long waits, many threads | Short critical sections |
| Implementation | OS-level syscall | Atomic hardware instruction |
| Fairness | Usually guaranteed by OS | Not guaranteed |

## Part 3: Key Concepts Summary

### Context Switch Fundamentals

A **context switch** saves the CPU state of the current thread and restores the CPU state of the next thread. In user-level threads:

1. **Save** callee-saved registers of the current thread
2. **Restore** callee-saved registers of the next thread
3. **`ret`** jumps to the restored `ra` (return address)

### Why Only Callee-Saved Registers?

The RISC-V ABI defines two categories:

```
Caller-saved (volatile):   ra*, a0-a7, t0-t6
  → Caller must save these before calling a function

Callee-saved (non-volatile): sp, s0-s11
  → Called function must restore these before returning
```

Since `thread_switch` is called like a regular function, the compiler already handles caller-saved registers before the call. We only need to explicitly save callee-saved registers.

> **⚠️ Warning:** `ra` is technically caller-saved BUT we must save it in the thread context because it holds **where the thread will return to** after `thread_switch` returns.

### Stack Layout for New Thread

```
+-------------------+  <- t->stack + STACK_SIZE  (ctx.sp points here)
|                   |  (stack grows downward →)
|   (empty stack)   |
|                   |
+-------------------+  <- t->stack[0]
```

## Part 5: Common Mistakes & Debugging

> **⚠️ Warning:** Stack pointer must point to the **top** (high address) not the bottom.

> **⚠️ Warning:** Forgetting `__sync_synchronize()` in spin locks can cause subtle bugs due to CPU/compiler reordering.

> **⚠️ Warning:** If you save caller-saved registers in `thread_switch`, it still works but wastes time and space.

## Part 6: Quick Reference — RISC-V Calling Convention

```
Registers:
  a0-a7  : function arguments / return values (caller-saved)
  t0-t6  : temporaries (caller-saved)
  s0-s11 : saved registers (callee-saved) ← save in thread_switch
  ra     : return address ← MUST save in thread context
  sp     : stack pointer ← MUST save in thread context
  gp     : global pointer
  tp     : thread pointer

Instructions used:
  sd reg, offset(base)  — store doubleword (64-bit) to memory
  ld reg, offset(base)  — load doubleword (64-bit) from memory
  ret                   — return (jumps to ra)
```
