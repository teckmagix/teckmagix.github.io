## Quiz 2 — Question 2: xv6 User-Level Thread Switching

## Overview

This lab involves implementing **user-level thread switching** in xv6. The key files are:

- `user/uthread.c` — User-level thread library (context, scheduler, thread create/yield)
- `user/uthread_switch.S` — Assembly routine to switch between thread contexts

The goal is to implement cooperative user-level threading (similar to how the kernel switches between processes, but entirely in user space).

---

## Background: How xv6 Kernel Context Switching Works

Before understanding the user-level version, it helps to understand the kernel's `swtch.S`:

```asm
# kernel/swtch.S
# void swtch(struct context *old, struct context *new);
# Save current registers into old, load from new.
```

The kernel saves **callee-saved registers** (`ra`, `sp`, `s0`–`s11`) into a `struct context`, then loads new ones. The `ra` register holds the return address — so when `swtch` returns, it returns into the *new* thread's execution point.

---

## The User-Level Thread Structure (`user/uthread.c`)

```c
#define STACK_SIZE  8192
#define MAX_THREAD  4

struct thread {
  char       stack[STACK_SIZE]; /* the thread's stack */
  int        state;             /* FREE, RUNNING, RUNNABLE */
  struct context ctx;           /* saved registers */
};
```

### Thread States

| State | Meaning |
|-------|---------|
| `FREE` | Slot not in use |
| `RUNNING` | Currently executing |
| `RUNNABLE` | Ready to run, not currently running |

## What You Need to Implement

### Part 1: `thread_create()` in `user/uthread.c`

When a new thread is created, you must set up its **initial context** so that when `thread_switch` is first called on it, execution starts at the given function.

```c
void thread_create(void (*func)())
{
  struct thread *t;

  for (t = all_thread; t < all_thread + MAX_THREAD; t++) {
    if (t->state == FREE) break;
  }
  t->state = RUNNABLE;

  // YOUR CODE HERE:
  // Set up t->ctx so that thread_switch will cause
  // the thread to start executing func().

  memset(&t->ctx, 0, sizeof(t->ctx));
  t->ctx.ra = (uint64)func;           // return address = function to run
  t->ctx.sp = (uint64)(t->stack + STACK_SIZE); // stack grows down
}
```

> **💡 Hint:** The stack pointer must point to the **top** of the stack (high address), because the stack grows downward on RISC-V. `t->stack + STACK_SIZE` is the address just past the end of the array — i.e., the top.

> **⚠️ Warning:** Do NOT point `sp` at `t->stack` (the bottom). That would cause immediate stack overflow on the first function call.

### Part 2: `uthread_switch.S` — The Context Switch Assembly

This is nearly identical to `kernel/swtch.S`. You save callee-saved registers from the current thread into its context struct, then load the new thread's registers.

```asm
# user/uthread_switch.S
# void thread_switch(struct context *old, struct context *new);
# Save current, restore next.

.text
.globl thread_switch
thread_switch:
    # Save callee-saved registers of current thread into old (a0)
    sd ra,   0(a0)
    sd sp,   8(a0)
    sd s0,  16(a0)
    sd s1,  24(a0)
    sd s2,  32(a0)
    sd s3,  40(a0)
    sd s4,  48(a0)
    sd s5,  56(a0)
    sd s6,  64(a0)
    sd s7,  72(a0)
    sd s8,  80(a0)
    sd s9,  88(a0)
    sd s10, 96(a0)
    sd s11,104(a0)

    # Restore callee-saved registers of new thread from new (a1)
    ld ra,   0(a1)
    ld sp,   8(a1)
    ld s0,  16(a1)
    ld s1,  24(a1)
    ld s2,  32(a1)
    ld s3,  40(a1)
    ld s4,  48(a1)
    ld s5,  56(a1)
    ld s6,  64(a1)
    ld s7,  72(a1)
    ld s8,  80(a1)
    ld s9,  88(a1)
    ld s10, 96(a1)
    ld s11,104(a1)

    ret   # returns to ra, which is the new thread's saved ra
```

> **💡 Hint:** The `ret` instruction jumps to whatever is in `ra`. For a **new** thread, `ra` was set to `func` in `thread_create`. For a **resuming** thread, `ra` was saved from the last call to `thread_switch`, so execution resumes after that call point.

## The `struct context` Layout

The offsets in the assembly above must match the C struct **exactly**:

```c
// In user/uthread.c (or a shared header)
struct context {
  uint64 ra;    // offset  0
  uint64 sp;    // offset  8
  uint64 s0;    // offset 16
  uint64 s1;    // offset 24
  uint64 s2;    // offset 32
  uint64 s3;    // offset 40
  uint64 s4;    // offset 48
  uint64 s5;    // offset 56
  uint64 s6;    // offset 64
  uint64 s7;    // offset 72
  uint64 s8;    // offset 80
  uint64 s9;    // offset 88
  uint64 s10;   // offset 96
  uint64 s11;   // offset 104
};
```

> **⚠️ Warning:** If your struct field order or padding differs, the offsets in the assembly will be wrong, causing crashes or undefined behavior. Always verify struct layout matches assembly offsets.

## How `thread_schedule()` Works (Already Provided)

```c
void thread_schedule(void)
{
  struct thread *t, *next_thread;

  // Find another RUNNABLE thread
  next_thread = 0;
  t = current_thread + 1;
  for (int i = 0; i < MAX_THREAD; i++) {
    if (t >= all_thread + MAX_THREAD)
      t = all_thread;
    if (t->state == RUNNABLE) {
      next_thread = t;
      break;
    }
    t = t + 1;
  }

  if (next_thread == 0) {
    printf("thread_schedule: no runnable threads\n");
    exit(-1);
  }

  if (current_thread != next_thread) {
    next_thread->state = RUNNING;
    t = current_thread;
    current_thread = next_thread;
    /* YOUR CODE: call thread_switch */
    thread_switch(&t->ctx, &current_thread->ctx);
  }
}
```

> **💡 Hint:** Notice that `thread_switch` is called **after** `current_thread` is updated. The first argument is the *old* thread's context (to save into), the second is the *new* thread's context (to load from).

## Why Only Callee-Saved Registers?

| Register Type | Saved By | Registers |
|--------------|----------|-----------|
| **Callee-saved** | The function being called (`thread_switch`) | `ra`, `sp`, `s0`–`s11` |
| **Caller-saved** | The calling function (already on stack) | `t0`–`t6`, `a0`–`a7` |

When a thread calls `thread_schedule()` → `thread_switch()`, the compiler already saved any caller-saved registers it needs onto the thread's **own stack**. Since each thread has its own stack (and `sp` is saved/restored), caller-saved registers are automatically preserved.

> **💡 Hint:** This is the same reasoning used in the xv6 kernel `swtch.S`. You only need to save what the **calling convention** says the callee must preserve.

## Execution Flow Walkthrough

### First time a new thread runs:

```
thread_schedule() called by Thread A
  → next_thread = Thread B (newly created)
  → thread_switch(&A->ctx, &B->ctx)
    → saves A's ra, sp, s0-s11 into A->ctx
    → loads B's ra (= func), sp (= top of B's stack) into registers
    → ret  →  jumps to func()
    → Thread B is now running func()
```

### Resuming a thread:

```
Thread B calls thread_yield() → thread_schedule()
  → thread_switch(&B->ctx, &C->ctx)
    → saves B's current ra (return address inside thread_schedule), sp, etc.
    → loads C's saved context
    → ret → resumes C right where it left off
```

## Common Mistakes

> **⚠️ Warning:** **Wrong stack pointer direction** — `t->ctx.sp = (uint64)t->stack` points to the *bottom* of the stack. Use `t->stack + STACK_SIZE` for the top.

> **⚠️ Warning:** **Forgetting to zero the context** — Without `memset(&t->ctx, 0, sizeof(t->ctx))`, leftover garbage values in `s0`–`s11` may cause crashes. Zeroing is safe since those registers don't matter for the initial call.

> **⚠️ Warning:** **Mismatched offsets** — If you add a field before `ra` in `struct context`, all your assembly offsets shift. Keep `ra` at offset 0.

> **⚠️ Warning:** **Not setting `state = RUNNABLE`** in `thread_create` — the scheduler won't pick up the thread.

## Comparison: Kernel vs. User-Level Threading

| Feature | Kernel (`swtch.S` / `proc.c`) | User-level (`uthread_switch.S` / `uthread.c`) |
|---------|-------------------------------|-----------------------------------------------|
| Context struct | `struct context` in `proc.h` | `struct context` in `uthread.c` |
| Stack | Kernel stack per process | User-allocated array per thread |
| Scheduler trigger | Timer interrupt + `yield()` | Explicit `thread_yield()` call (cooperative) |
| Saved registers | `ra`, `sp`, `s0`–`s11` | Same: `ra`, `sp`, `s0`–`s11` |
| Assembly file | `kernel/swtch.S` | `user/uthread_switch.S` |
| Parallelism | Yes (multiple CPUs) | No (single process, cooperative) |
