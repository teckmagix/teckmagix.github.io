## Core Concepts: User-Level Threading in xv6

### What is a User-Level Thread?

A **user-level thread** is a thread managed entirely in user space, without kernel involvement for scheduling. The kernel sees only a single process; the user-level thread library handles:

1. **Thread creation** — allocating a stack and setting up initial context
2. **Context switching** — saving/restoring registers between threads
3. **Scheduling** — deciding which thread runs next

> **💡 Hint:** User-level threads are faster to create and switch than kernel threads because no system call is needed, but they cannot run truly in parallel on multiple CPUs.

## Key File: `user/uthread.c` Analysis

### Thread Structure

```c
struct thread {
  char       stack[STACK_SIZE]; /* the thread's stack */
  int        state;             /* FREE, RUNNING, RUNNABLE */
  /* TODO: you need to add a context here */
};
```

The thread structure holds:
- A **fixed-size stack** for the thread
- A **state** (FREE, RUNNING, RUNNABLE)
- A **context** (registers to save/restore during context switch)

### The Context Structure

You need to add a context struct similar to the kernel's `struct context` in `kernel/proc.h`:

```c
struct thread_context {
  uint64 ra;   // return address
  uint64 sp;   // stack pointer
  // callee-saved registers
  uint64 s0;
  uint64 s1;
  uint64 s2;
  uint64 s3;
  uint64 s4;
  uint64 s5;
  uint64 s6;
  uint64 s7;
  uint64 s8;
  uint64 s9;
  uint64 s10;
  uint64 s11;

```

> **💡 Hint:** Only **callee-saved registers** need to be saved. Caller-saved registers are saved by the calling convention on the stack automatically.

### Thread Creation — `thread_create()`

```c
void 
thread_create(void (*func)())
{
  struct thread *t;

  for (t = all_thread; t < all_thread + MAX_THREAD; t++) {
    if (t->state == FREE) break;
  }
  t->state = RUNNABLE;
  // YOUR CODE HERE
}
```

**What you need to do:**

1. Set up the thread's **stack pointer** (`sp`) to point to the **top** of the thread's stack (stacks grow downward)
2. Set the **return address** (`ra`) to the function `func` so when the thread is first scheduled, it starts executing `func`

```c
// Correct implementation:
t->context.sp = (uint64)(t->stack + STACK_SIZE); // top of stack
t->context.ra = (uint64)func;                     // entry point
```

> **⚠️ Warning:** The stack grows **downward** in RISC-V (and most architectures). The `sp` must point to the **end** of the stack array (highest address), not the beginning.

### Thread Scheduler — `thread_schedule()`

```c
void 
thread_schedule(void)
{
  struct thread *t, *next_thread;

  /* Find another runnable thread. */
  next_thread = 0;
  t = current_thread + 1;
  for(int i = 0; i < MAX_THREAD; i++){
    if(t >= all_thread + MAX_THREAD)
      t = all_thread;
    if(t->state == RUNNABLE) {
      next_thread = t;
      break;
    }
    t = t + 1;
  }

  if (next_thread == 0) {
    printf("thread_schedule: no runnable threads\n");
    exit(-1);
  }

  if (current_thread != next_thread) {
    next_thread->state = RUNNING;
    t = current_thread;
    current_thread = next_thread;
    /* YOUR CODE HERE: invoke thread_switch */
    thread_switch((uint64)&t->context, (uint64)&current_thread->context);
  }
}
```

**Key insight:** `thread_switch` saves the current context and restores the next thread's context.

## Key File: `user/uthread_switch.S` Analysis

### The Context Switch Assembly

This is the most critical piece. The `thread_switch` function must:
1. **Save** all callee-saved registers of the **current** thread into its context struct
2. **Restore** all callee-saved registers of the **next** thread from its context struct

```asm
	.text

	/*
         * save the old thread's registers,
         * restore the new thread's registers.
         */

	.globl thread_switch
thread_switch:
	/* YOUR CODE HERE */
	ret    /* return to ra */
```

### Complete Implementation

```asm
	.text
	.globl thread_switch
thread_switch:
    /* Save callee-saved registers of current thread */
    sd ra,   0(a0)
    sd sp,   8(a0)
    sd s0,  16(a0)
    sd s1,  24(a0)
    sd s2,  32(a0)
    sd s3,  40(a0)
    sd s4,  48(a0)
    sd s5,  56(a0)
    sd s6,  64(a0)
    sd s7,  72(a0)
    sd s8,  80(a0)
    sd s9,  88(a0)
    sd s10, 96(a0)
    sd s11,104(a0)

    /* Restore callee-saved registers of next thread */
    ld ra,   0(a1)
    ld sp,   8(a1)
    ld s0,  16(a1)
    ld s1,  24(a1)
    ld s2,  32(a1)
    ld s3,  40(a1)
    ld s4,  48(a1)
    ld s5,  56(a1)
    ld s6,  64(a1)
    ld s7,  72(a1)
    ld s8,  80(a1)
    ld s9,  88(a1)
    ld s10, 96(a1)
    ld s11,104(a1)

    ret
```

**Function signature:** `thread_switch(uint64 old_context, uint64 new_context)`
- `a0` = pointer to **old** thread's context (save here)
- `a1` = pointer to **new** thread's context (restore from here)
- `ret` jumps to `ra` — which is the new thread's saved `ra` (its entry point or where it last called `thread_schedule`)

> **💡 Hint:** The `sd` instruction stores a 64-bit double-word. The `ld` instruction loads a 64-bit double-word. The offsets (0, 8, 16, ...) correspond to the byte offsets of each field in `struct thread_context`.

## Register Conventions (RISC-V)

| Register | ABI Name | Description | Saved by |
|----------|----------|-------------|----------|
| x1 | ra | Return address | Caller |
| x2 | sp | Stack pointer | Callee |
| x8 | s0/fp | Saved reg / frame ptr | Callee |
| x9 | s1 | Saved register | Callee |
| x18-x27 | s2-s11 | Saved registers | Callee |
| x10-x11 | a0-a1 | Function arguments | Caller |
| x12-x17 | a2-a7 | Function arguments | Caller |
| x5-x7 | t0-t2 | Temporaries | Caller |
| x28-x31 | t3-t6 | Temporaries | Caller |

> **⚠️ Warning:** Only **callee-saved** registers (s0–s11, sp, ra) need explicit saving in `thread_switch`. Caller-saved registers (t0-t6, a0-a7) are the caller's responsibility and are already saved on the stack if needed.

### Typical Questions About pthread_locks

#### Why is there contention?

When multiple threads try to **acquire the same lock** simultaneously, only one succeeds — the others **spin or block**, wasting CPU cycles.

#### What is a spin lock vs. a mutex?

| Feature | Spin Lock | Mutex (Sleeping Lock) |
|---------|-----------|----------------------|
| Waiting behavior | Busy-waits (loops) | Sleeps (yields CPU) |
| Good for | Short critical sections | Long critical sections |
| CPU usage while waiting | High | Low |
| Overhead to acquire | Low (no syscall) | Higher (may syscall) |

## How the First Thread Switch Works

```
thread_create(func)
  → sets ra = func
  → sets sp = top of stack

thread_schedule()
  → finds RUNNABLE thread
  → calls thread_switch(old_ctx, new_ctx)

thread_switch
  → saves old thread's registers to old_ctx
  → loads new thread's registers from new_ctx
  → ret  (jumps to new_ctx->ra = func)

func() starts executing!
```

When `func()` calls `thread_schedule()` again later:
```
thread_schedule() → thread_switch()
  → saves current thread's ra (= return address inside thread_schedule)
  → restores next thread's context
  → ret  (jumps to next thread's saved ra)
```

> **💡 Hint:** The key insight is that `ret` in `thread_switch` jumps to whatever is in `ra` **after** the load. For a brand-new thread, `ra` = the function to run. For a previously-suspended thread, `ra` = the return address from the last call to `thread_switch` (inside `thread_schedule`).

## Common Mistakes and How to Avoid Them

> **⚠️ Warning:** **Stack direction error** — Setting `sp = t->stack` (bottom) instead of `sp = t->stack + STACK_SIZE` (top) will cause immediate stack corruption when the thread tries to push anything.

> **⚠️ Warning:** **Wrong context offsets** — If your `struct thread_context` field order doesn't match the offsets in your assembly (`sd ra, 0(a0)`, etc.), registers will be saved/restored to wrong locations.

> **⚠️ Warning:** **Saving caller-saved registers** — You do NOT need to save `t0-t6` or `a0-a7` in `thread_switch`. The RISC-V calling convention says the caller is responsible for these.

> **⚠️ Warning:** **Forgetting `memset`** — Always zero-initialize the context in `thread_create`. Garbage values in `s0-s11` won't crash things immediately but can cause subtle bugs.

## Comparison: User Threads vs Kernel Threads

| Feature | User Threads (uthread) | Kernel Threads (xv6 proc) |
|---------|----------------------|--------------------------|
| Context switch mechanism | `thread_switch` in user space | `swtch` in kernel |
| Registers saved | ra, sp, s0-s11 | ra, sp, s0-s11 |
| Switch triggers | Cooperative (explicit yield) | Preemptive (timer interrupt) |
| Parallelism | No (single CPU process) | Yes (multiple CPUs) |
| Switch cost | Very low | Low (no user/kernel transition) |
| Protection | None between threads | Full isolation |

## Key Kernel Reference: `kernel/swtch.S`

The kernel's context switch is nearly identical in structure:

```asm
# kernel/swtch.S - kernel context switch
.globl swtch
swtch:
    sd ra,    0(a0)
    sd sp,    8(a0)
    sd s0,   16(a0)
    # ... saves s0-s11 to struct context at a0
    
    ld ra,    0(a1)
    ld sp,    8(a1)
    ld s0,   16(a1)
    # ... restores s0-s11 from struct context at a1
    ret
```

Your `uthread_switch.S` should mirror this exactly, using the same register offsets.
