## Quiz 2 - Question 2: xv6 Thread Switching (uthread)

## Overview

This lab involves implementing **user-level thread switching** in xv6. The key files are:
- `user/uthread.c` — user-level thread library (needs completion)
- `user/uthread_switch.S` — assembly context switch routine (needs completion)

---

## Understanding the Problem

### What Is User-Level Thread Switching?

In this assignment, you implement a **cooperative** (non-preemptive) user-level threading system. Threads yield control voluntarily. The OS knows nothing about these threads — they all share a single OS process.

> **💡 Hint:** Think of it like `setjmp`/`longjmp` — you save registers, swap stacks, restore registers.

### Key Concepts

| Concept | Description |
|--------|-------------|
| **Context** | Set of register values (saved state) of a thread |
| **Stack** | Each thread needs its own private stack |
| **Switch** | Save current thread's context → restore next thread's context |
| **PC (Program Counter)** | Where execution resumes after switch |
| **SP (Stack Pointer)** | Points to current thread's stack |

---

## Files to Modify

### 1. `user/uthread.c`

This file defines:
- `struct thread` — thread state (stack, context, status)
- `thread_create()` — initializes a new thread
- `thread_schedule()` — picks the next runnable thread and switches to it
- `thread_yield()` — called by a thread to give up the CPU

### 2. `user/uthread_switch.S`

Assembly file that performs the **actual register save/restore**.

## Step-by-Step Solution

### Step 1: Define the Thread Context Structure

In `user/uthread.c`, you need a structure to save/restore CPU registers. For RISC-V, you must save **callee-saved registers** (the ABI says the caller saves caller-saved regs before calling a function — we only need to preserve what the callee is responsible for).

RISC-V callee-saved registers: `ra`, `sp`, `s0`–`s11`

```c
// Add this struct to uthread.c (or uthread.h if referenced from .S)
struct thread_context {
    uint64 ra;   // return address
    uint64 sp;   // stack pointer
    // callee-saved registers
    uint64 s0;
    uint64 s1;
    uint64 s2;
    uint64 s3;
    uint64 s4;
    uint64 s5;
    uint64 s6;
    uint64 s7;
    uint64 s8;
    uint64 s9;
    uint64 s10;
    uint64 s11;
};
```

> **⚠️ Warning:** The layout of this struct must **exactly match** the offsets used in `uthread_switch.S`. If you reorder fields, the assembly will save/restore wrong registers!

### Step 2: Add Context to the Thread Struct

The `struct thread` in `uthread.c` needs to include the context:

```c
struct thread {
    char            stack[STACK_SIZE];   // thread's stack
    int             state;               // FREE, RUNNING, or RUNNABLE
    struct thread_context context;       // saved registers
};
```

### Step 3: Implement `thread_create()`

When creating a thread, you must:
1. Find a free thread slot
2. Mark it as `RUNNABLE`
3. Set up its stack so that when it is first scheduled, it starts executing `func`

The trick: set `ra` (return address) to `func`, and set `sp` to the **top** of the thread's stack (stacks grow downward).

```c
void thread_create(void (*func)())
{
    struct thread *t;

    for (t = all_thread; t < all_thread + MAX_THREAD; t++) {
        if (t->state == FREE) break;
    }

    t->state = RUNNABLE;

    // Set the return address to the thread function
    t->context.ra = (uint64)func;

    // Stack grows downward: sp points to the TOP (high address) of the stack
    t->context.sp = (uint64)(t->stack + STACK_SIZE);
}
```

> **💡 Hint:** `t->stack + STACK_SIZE` points to the byte just past the end of the stack array — this is the initial stack top because RISC-V stacks grow downward.

> **⚠️ Warning:** Do NOT set `sp` to `t->stack` (the bottom). That would cause an immediate stack overflow. Use `t->stack + STACK_SIZE`.

### Step 4: Implement `thread_schedule()`

This function selects the next `RUNNABLE` thread and calls `thread_switch` to perform the context switch:

```c
void thread_schedule(void)
{
    struct thread *t, *next_thread;

    /* Find another runnable thread. */
    next_thread = 0;
    t = current_thread + 1;
    for (int i = 0; i < MAX_THREAD; i++) {
        if (t >= all_thread + MAX_THREAD)
            t = all_thread;
        if (t->state == RUNNABLE) {
            next_thread = t;
            break;
        }
        t = t + 1;
    }

    if (next_thread == 0) {
        printf("thread_schedule: no runnable threads\n");
        exit(-1);
    }

    if (current_thread != next_thread) {
        next_thread->state = RUNNING;
        t = current_thread;
        current_thread = next_thread;
        // THE ACTUAL SWITCH:
        thread_switch((uint64)&t->context, (uint64)&current_thread->context);
    }
}
```

### Step 5: Implement `uthread_switch.S`

This is the most critical piece. It must:
1. Save all callee-saved registers of the **current** (old) thread into `a0` (pointer to old context)
2. Restore all callee-saved registers of the **next** (new) thread from `a1` (pointer to new context)
3. Return (`ret`) — which jumps to `ra`, which was set to the new thread's function (or wherever it left off)

```asm
	.text

/*
 * void thread_switch(uint64 old_context_addr, uint64 new_context_addr)
 * a0 = address of old thread's context struct
 * a1 = address of new thread's context struct
 *
 * Context struct layout (must match struct thread_context):
 *   offset  0: ra
 *   offset  8: sp
 *   offset 16: s0
 *   offset 24: s1
 *   offset 32: s2
 *   offset 40: s3
 *   offset 48: s4
 *   offset 56: s5
 *   offset 64: s6
 *   offset 72: s7
 *   offset 80: s8
 *   offset 88: s9
 *   offset 96: s10
 *   offset 104: s11
 */
	.globl thread_switch
thread_switch:
	/* Save callee-saved registers of the OLD thread into *a0 */
	sd ra,    0(a0)
	sd sp,    8(a0)
	sd s0,   16(a0)
	sd s1,   24(a0)
	sd s2,   32(a0)
	sd s3,   40(a0)
	sd s4,   48(a0)
	sd s5,   56(a0)
	sd s6,   64(a0)
	sd s7,   72(a0)
	sd s8,   80(a0)
	sd s9,   88(a0)
	sd s10,  96(a0)
	sd s11, 104(a0)

	/* Restore callee-saved registers of the NEW thread from *a1 */
	ld ra,    0(a1)
	ld sp,    8(a1)
	ld s0,   16(a1)
	ld s1,   24(a1)
	ld s2,   32(a1)
	ld s3,   40(a1)
	ld s4,   48(a1)
	ld s5,   56(a1)
	ld s6,   64(a1)
	ld s7,   72(a1)
	ld s8,   80(a1)
	ld s9,   88(a1)
	ld s10,  96(a1)
	ld s11, 104(a1)

	ret   /* jump to ra of new thread */
```

> **⚠️ Warning:** The byte offsets (0, 8, 16, ...) must **exactly match** the layout of `struct thread_context`. Each `uint64` is 8 bytes. If you add or reorder fields in the struct, update the offsets here too.

> **💡 Hint:** `ret` is a pseudoinstruction for `jalr x0, ra, 0` — it jumps to the address in `ra`. Since we loaded the new thread's `ra` (which is either the thread function or the saved return address from a previous yield), execution continues there.

## How It All Works Together

```
thread_create(func):
    - Sets context.ra = func
    - Sets context.sp = top of stack
    - Sets state = RUNNABLE

thread_yield():
    - Sets current_thread->state = RUNNABLE
    - Calls thread_schedule()

thread_schedule():
    - Finds next RUNNABLE thread
    - Calls thread_switch(&old->context, &new->context)

thread_switch (assembly):
    - Saves ra, sp, s0-s11 → old->context
    - Loads ra, sp, s0-s11 ← new->context
    - ret → jumps to new thread's ra
```

### First-Time Scheduling of a New Thread

When `thread_switch` does `ret` for a **brand new** thread:
- `ra` = `func` (set in `thread_create`)
- `sp` = top of new thread's stack
- Result: execution jumps to `func` with a fresh stack ✓

### Resuming a Yielded Thread

When `thread_switch` does `ret` for a **previously yielded** thread:
- `ra` = the return address **inside** `thread_switch` at the call site in `thread_schedule`
- `sp` = the thread's saved stack pointer
- Result: execution returns back into `thread_schedule`, which returns to `thread_yield`, which returns to wherever the thread called `thread_yield` ✓

## Common Mistakes

> **⚠️ Warning:** **Wrong stack direction** — Setting `sp = t->stack` instead of `t->stack + STACK_SIZE`. Stacks grow *downward*, so the initial SP must be at the *high* end.

> **⚠️ Warning:** **Saving wrong registers** — You only need to save callee-saved registers (`ra, sp, s0–s11`). Caller-saved registers (`a0–a7, t0–t6`) are the caller's responsibility and don't need to be in the context.

> **⚠️ Warning:** **Struct/assembly offset mismatch** — If the struct has padding or different field order than what the assembly assumes, the switch will corrupt register values.

> **⚠️ Warning:** **Not saving `ra`** — Without saving `ra`, the thread can never resume where it left off. It's arguably the most important register to save.

## Register Reference Table

| Register | ABI Name | Role in Switch |
|----------|----------|----------------|
| `x1` | `ra` | Return address — where thread resumes |
| `x2` | `sp` | Stack pointer — thread's own stack |
| `x8` | `s0/fp` | Callee-saved / frame pointer |
| `x9` | `s1` | Callee-saved |
| `x18–x27` | `s2–s11` | Callee-saved |
| `x10–x17` | `a0–a7` | **Not saved** (caller-saved, args) |
| `x5–x7,x28–x31` | `t0–t6` | **Not saved** (caller-saved, temporaries) |

## Comparison: User Thread vs Kernel Thread Switch

| Feature | User Thread (`uthread`) | Kernel Thread (`swtch`) |
|---------|------------------------|------------------------|
| Privilege level | User space | Kernel space |
| Register save | Callee-saved only | Callee-saved only |
| Struct name | `thread_context` | `context` |
| Where defined | `uthread.c` / `uthread_switch.S` | `proc.h` / `swtch.S` |
| Scheduler | `thread_schedule()` | `scheduler()` in `proc.c` |
| Preemptive? | No (cooperative) | Yes (timer interrupts) |

> **💡 Hint:** The kernel's `swtch.S` is almost identical to `uthread_switch.S`. If you understand one, you understand the other!
