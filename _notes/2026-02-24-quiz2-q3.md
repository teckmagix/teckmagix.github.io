## Part 1: Understanding the Thread Data Structures

### `user/uthread.h` — Thread Header

The header defines the core structures for user-level threading:

```c
// Thread states
#define FREE        0   // Thread slot is unused
#define RUNNING     1   // Thread is currently running
#define RUNNABLE    2   // Thread is ready to run
```

A **thread** has:
- A **stack** (fixed-size array of bytes)
- A **state** (FREE, RUNNING, or RUNNABLE)
- A **saved context** (registers saved during a context switch)

```c
#define STACK_SIZE  8192   // 8KB stack per thread
#define MAX_THREAD  4      // Maximum 4 threads simultaneously
```

> **💡 Hint:** The context structure must save all **callee-saved registers** per the RISC-V calling convention. These are: `ra`, `sp`, `s0`–`s11`.

## Part 2: The Context Switch — `user/uthread_switch.S`

### What is a Context Switch?

A context switch saves the current thread's CPU register state and restores another thread's saved register state. This is what makes "switching" between threads possible.

### RISC-V Calling Convention — Callee-Saved Registers

The following registers **must be saved** across a function call (callee-saved):

| Register | ABI Name | Role |
|----------|----------|------|
| `x1` | `ra` | Return address |
| `x2` | `sp` | Stack pointer |
| `x8` | `s0/fp` | Saved register / frame pointer |
| `x9` | `s1` | Saved register |
| `x18`–`x27` | `s2`–`s11` | Saved registers |

### The `thread_switch` Function

The assembly function `thread_switch(struct context *old, struct context *new)` must:
1. **Save** caller's registers into `old` context
2. **Restore** registers from `new` context

```asm
# user/uthread_switch.S
.text

/*
 * void thread_switch(struct context *old, struct context *new);
 *
 * a0 = pointer to old context (save here)
 * a1 = pointer to new context (restore from here)
 */
.globl thread_switch
thread_switch:
    /* Save callee-saved registers into old context (a0) */
    sd ra,   0(a0)
    sd sp,   8(a0)
    sd s0,  16(a0)
    sd s1,  24(a0)
    sd s2,  32(a0)
    sd s3,  40(a0)
    sd s4,  48(a0)
    sd s5,  56(a0)
    sd s6,  64(a0)
    sd s7,  72(a0)
    sd s8,  80(a0)
    sd s9,  88(a0)
    sd s10, 96(a0)
    sd s11,104(a0)

    /* Restore callee-saved registers from new context (a1) */
    ld ra,   0(a1)
    ld sp,   8(a1)
    ld s0,  16(a1)
    ld s1,  24(a1)
    ld s2,  32(a1)
    ld s3,  40(a1)
    ld s4,  48(a1)
    ld s5,  56(a1)
    ld s6,  64(a1)
    ld s7,  72(a1)
    ld s8,  80(a1)
    ld s9,  88(a1)
    ld s10, 96(a1)
    ld s11,104(a1)

    ret   /* Return to the new thread (jumps to ra) */
```

> **⚠️ Warning:** You only save **callee-saved** registers. Caller-saved registers (`t0`–`t6`, `a0`–`a7`) do NOT need to be saved because by calling convention, the calling function already saved them if it needs them.

## Part 3: The Thread Implementation — `user/uthread.c`

### Key Functions You Must Implement

#### 3.1 `thread_create(void (*func)())`

Creates a new thread that will execute `func`.

**What it must do:**
1. Find a FREE thread slot
2. Set its state to RUNNABLE
3. Set up the stack so that when the thread is first scheduled, it starts executing `func`
4. The key trick: set `context.ra` to the function address, and `context.sp` to the top of the stack

```c
void thread_create(void (*func)())
{
    struct thread *t;

    for (t = all_thread; t < all_thread + MAX_THREAD; t++) {
        if (t->state == FREE)
            break;
    }

    t->state = RUNNABLE;

    // Set up context so first switch jumps to func
    // ra = function to run when thread first scheduled
    t->context.ra = (uint64) func;

    // sp = top of this thread's stack
    // Stack grows downward, so sp = stack + STACK_SIZE
    t->context.sp = (uint64) t->stack + STACK_SIZE;
}
```

> **💡 Hint:** The stack grows **downward** in memory. So the initial stack pointer should point to `stack + STACK_SIZE` (the **top** of the allocated array), not `stack[0]`.

> **⚠️ Warning:** If you set `ra` to `func` and then `ret` is executed in `thread_switch`, execution jumps to `func`. This is how the thread "starts". Make sure `sp` is properly aligned (16-byte aligned on RISC-V).

#### 3.2 `thread_schedule()`

The scheduler picks the next RUNNABLE thread and switches to it.

```c
void thread_schedule(void)
{
    struct thread *t, *next_thread;

    /* Find another runnable thread */
    next_thread = 0;
    t = current_thread + 1;
    for (int i = 0; i < MAX_THREAD; i++) {
        if (t >= all_thread + MAX_THREAD)
            t = all_thread;
        if (t->state == RUNNABLE) {
            next_thread = t;
            break;
        }
        t = t + 1;
    }

    if (next_thread == 0) {
        printf("thread_schedule: no runnable threads\n");
        exit(-1);
    }

    if (current_thread != next_thread) {
        next_thread->state = RUNNING;
        t = current_thread;
        current_thread = next_thread;
        /* Switch context: save old, restore new */
        thread_switch(&t->context, &current_thread->context);
    } else {
        next_thread->state = RUNNING;
    }
}
```

#### 3.3 `thread_yield()`

Gives up the CPU voluntarily.

```c
void thread_yield(void)
{
    current_thread->state = RUNNABLE;
    thread_schedule();
}
```

## Part 4: The Context Structure

The context struct must match the layout assumed in `uthread_switch.S`. Here is what it should look like in `uthread.h`:

```c
struct context {
    uint64 ra;    // offset  0
    uint64 sp;    // offset  8
    uint64 s0;    // offset 16
    uint64 s1;    // offset 24
    uint64 s2;    // offset 32
    uint64 s3;    // offset 40
    uint64 s4;    // offset 48
    uint64 s5;    // offset 56
    uint64 s6;    // offset 64
    uint64 s7;    // offset 72
    uint64 s8;    // offset 80
    uint64 s9;    // offset 88
    uint64 s10;   // offset 96
    uint64 s11;   // offset 104
};

struct thread {
    char            stack[STACK_SIZE];  // Thread's stack
    int             state;              // FREE, RUNNING, or RUNNABLE
    struct context  context;            // Saved registers
};
```

> **⚠️ Warning:** The byte offsets in the struct **must exactly match** the offsets used in the assembly (`sd ra, 0(a0)`, `sd sp, 8(a0)`, etc.). If the struct layout differs, the wrong memory is read/written.

## Part 5: How Thread Switching Works — Step by Step

```
Thread A (running)       thread_yield()
     |                        |
     v                        v
  current->state = RUNNABLE
     |
     v
  thread_schedule() picks Thread B
     |
     v
  thread_switch(&A->context, &B->context)
     |                         |
     v                         v
  Save A's registers       Restore B's registers
  into A->context          from B->context
     |
     v
  ret  --> jumps to B->context.ra
           (which is either B's func if first time,
            or the return address inside thread_switch
            from when B last called thread_switch)
```

### First-time Thread Execution

When a thread runs for the **first time**:
- `context.ra` = address of the thread's function (`func`)
- `context.sp` = top of thread's stack
- `ret` in `thread_switch` loads `ra` and jumps to `func`
- `sp` is properly set so the function has a valid stack

### Subsequent Thread Execution

When a thread resumes after yielding:
- `context.ra` = return address **inside** `thread_switch` (the point after the call)
- `ret` jumps back into `thread_schedule` → returns to `thread_yield` → returns to wherever the thread was running

## Part 6: Comparison with Kernel-Level Threads

| Feature | User-Level Threads (this lab) | Kernel Threads (xv6 `proc.c`) |
|---------|-------------------------------|-------------------------------|
| Context switch location | `user/uthread_switch.S` | `kernel/swtch.S` |
| Registers saved | `ra, sp, s0–s11` | `ra, sp, s0–s11` |
| State management | `uthread.c` | `proc.c` |
| Scheduler | Round-robin in user space | Kernel scheduler |
| Blocking I/O | Blocks all threads | Blocks only that process |
| Preemption | Cooperative only | Timer-interrupt preemptive |

> **💡 Hint:** Notice that `kernel/swtch.S` and `user/uthread_switch.S` are essentially the **same** code pattern. The kernel uses the same save/restore technique for switching between kernel threads.

## Part 7: Example Tests Walkthrough

### `user/uthread_ex0.c` — Basic Test

```c
// Typical pattern:
void thread_a(void) {
    for (int i = 0; i < 100; i++) {
        printf("thread_a %d\n", i);
        thread_yield();  // Give up CPU
    }
    thread_exit();
}

void thread_b(void) {
    for (int i = 0; i < 100; i++) {
        printf("thread_b %d\n", i);
        thread_yield();
    }
    thread_exit();
}

int main(void) {
    thread_init();
    thread_create(thread_a);
    thread_create(thread_b);
    thread_schedule();  // Start scheduling
    exit(0);
}
```

Expected output interleaves `thread_a` and `thread_b` prints, demonstrating cooperative multithreading.

## Part 8: Common Mistakes and Debugging Tips

> **⚠️ Warning: Stack Direction**
> The stack grows **downward**. Always initialize `sp = stack + STACK_SIZE`, not `sp = stack`. If you use `sp = stack`, the thread will corrupt memory immediately.

> **⚠️ Warning: Context Struct Alignment**
> The `context` struct fields must be in exactly the order matching the assembly offsets. If you add fields or reorder them, the assembly will save/restore wrong values.

> **⚠️ Warning: Missing `ret` in Assembly**
> The `thread_switch` function must end with `ret`. Without it, execution falls through to garbage memory.

> **⚠️ Warning: Caller vs Callee Saved**
> Do NOT save `a0`–`a7` or `t0`–`t6`. These are caller-saved and the C compiler handles them automatically. Saving them wastes space and can introduce bugs.

> **💡 Hint:** If your threads crash immediately, check:
> 1. Is `context.sp` pointing to the **top** of the stack?
> 2. Is `context.ra` set to the correct function?
> 3. Are the assembly offsets matching the C struct layout?
